# Ailux Agent UI 规范

**版本**：v1.0  
**适用范围**：Ailux Agent 桌面端四模块工作台界面  
**文档目的**：统一当前前端原型在视觉、布局、组件与交互层面的设计标准，确保后续设计迭代、前端实现和多页面扩展保持一致性。

## 1. 设计目标与界面定位

Ailux Agent 的界面不是通用型聊天产品，而是面向生物医药研究任务的 **智能体工作台**。因此，界面的核心目标不是营造强装饰性的品牌展示，而是让用户在进入页面后的数秒内，立即理解页面由哪四个区域组成、当前任务处于什么阶段、应该从哪里发起对话，以及结果最终沉淀在哪里。

当前版本采用 **四模块结构**，分别为左侧任务列表、中部 Agent 对话区域、右上 Plan 计划区域和右下 Result 结果文件区域。该结构要求所有新增设计都服从明确的信息层级，避免将说明型内容、营销型视觉和装饰性模块插入到核心工作流中，破坏使用路径。

> **界面原则**：先识别任务，再理解过程，最后消费结果。

## 2. 信息架构规范

界面信息架构应以“任务上下文—对话主流程—执行状态—结果沉淀”为主线组织。四个核心模块之间的优先级不应等权，而应由左到右、由主到辅逐层递减。

| 模块 | 位置 | 主要作用 | 信息优先级 | 设计要求 |
|---|---|---:|---:|---|
| 任务列表 | 左侧固定栏 | 承载任务切换、任务上下文与新建入口 | 高 | 保持轻量、稳定、可快速扫描 |
| Agent 对话区域 | 页面中心 | 作为主工作区，承担提问、回复与继续追问 | 最高 | 视觉面积最大，输入与消息阅读优先 |
| Plan 计划区域 | 右上 | 呈现执行步骤、进度和状态 | 中 | 强调过程透明，但不能压过对话区 |
| Result 结果文件区域 | 右下 | 收纳输出文件并支持搜索与访问 | 中 | 信息密度高于 Plan，但视觉存在感更弱 |

在任何后续页面中，若需要增加新能力，应优先将其归入这四个模块中的某一类，而不是新增独立的大模块。例如，“任务参数配置”可作为任务列表的二级面板，“图表预览”可作为 Result 的展开态，“Agent 思考过程”可作为对话区中的系统消息，而不应直接形成第五块平级区域。

## 3. 视觉风格总则

当前界面应保持 **克制、专业、偏产品化** 的科研工作台气质。视觉感受应当建立在蓝色品牌体系之上，同时辅以少量浅桃色作为执行中状态的强调色。整体风格应体现轻量层次、柔和边界与高可读性，而不是厚重、大面积高饱和背景。

从风格定位上看，Ailux Agent 应呈现三种并存特征：其一是 **科研感**，通过冷色系、规整排布、清晰信息层级来体现；其二是 **智能体感**，通过对话结构、步骤进度和状态标签来体现；其三是 **产品感**，通过收敛字号、圆角、阴影和边框厚度来体现。三者需要平衡，任何一项过强都会让界面走偏。

## 4. 品牌色与语义色规范

当前版本的界面色彩已经基本建立起以深蓝为主、亮蓝为强调、浅紫为过渡、浅桃为状态提示的配色逻辑。后续设计与开发应在此框架内延展，而不应新增无来源的高饱和颜色。

| 颜色名称 | HEX | 角色定义 | 推荐用途 |
|---|---|---|---|
| Discovery Blue | `#070261` | 品牌主深色 | 主标题、重点标题、深色品牌识别 |
| Antibody Indigo | `#161FAD` | 主操作色 | 主按钮、选中状态、进度条起始色 |
| Bio Blue | `#1724D8` | 强调色 | 用户消息气泡、重要动作 Hover、操作强化 |
| Molecular Blue | `#1928FB` | 高亮延展色 | 渐变尾端、局部高亮、少量强调 |
| Petri Dish Lilac | `#848CFE` | 过渡辅助色 | 进度渐变、浅色高亮背景、选择反馈 |
| Cell Culture Peach | `#FFC997` | 暖色提示 | “进行中”状态、轻提醒、局部强调 |
| Lab Coat White | `#FFFFFF` | 主背景色 | 面板背景、输入底色、卡片底色 |
| Copy Gray | `#A6A6A6` | 次级文字参考色 | 元信息、时间、辅助说明 |
| Midnight | `#000012` | 极深对比色 | 特殊深底场景，当前版本少用 |

### 4.1 色彩使用比例

为了确保界面维持专业感，建议后续页面保持以下近似比例。

| 色彩类别 | 建议占比 | 说明 |
|---|---:|---|
| 白色与近白背景 | 65%–75% | 维持工作台的清洁度与可读性 |
| 浅灰蓝中性色 | 15%–20% | 用于面板、边界、分区与输入背景 |
| 品牌蓝色系 | 10%–15% | 用于标题、按钮、选中状态与进度强调 |
| 暖色提示色 | 3%–5% | 仅用于“进行中”等状态提醒 |

### 4.2 语义状态色规则

| 语义状态 | 背景 | 文字 | 边框 | 使用说明 |
|---|---|---|---|---|
| 默认 | `#F8FAFF` 近似 | 深灰蓝 | 浅边框 | 普通卡片、普通列表项 |
| 选中 | 蓝紫浅底 | `#070261` / `#161FAD` | 蓝色浅边框 | 当前任务、当前 Tab、当前聚焦区域 |
| Hover | 白底略提亮 | 深灰蓝 | 边框加深 | 不使用剧烈放大或强投影 |
| 进行中 | 浅桃底 | 棕橙深字 | 桃色淡边框 | Plan 状态标签、进度提示 |
| 完成 | 浅蓝底 | `#161FAD` | 蓝色淡边框 | 已完成步骤、成功提示 |
| 待执行 | 浅灰底 | 中灰文字 | 灰色边框 | 尚未执行的任务步骤 |

## 5. 字体与排版规范

当前版本已经将字体收敛为两组：**IBM Plex Sans** 负责正文与界面信息，**Space Grotesk** 负责标题与模块标识。该搭配能够兼顾科研产品的理性气质与品牌辨识度，应保持一致，不建议替换为更具装饰性的字体。

| 字体角色 | 字体 | 使用范围 | 原则 |
|---|---|---|---|
| 正文字体 | IBM Plex Sans | 正文、列表、输入框、状态、文件名 | 保证高可读性与长时间阅读舒适度 |
| 标题字体 | Space Grotesk | 页面主标题、模块标题 | 控制使用频次，避免全界面过度展示化 |

### 5.1 建议字号层级

以下为基于当前版本提炼的桌面端建议层级，后续页面应尽量复用，不建议任意新增大量中间字号。

| 层级 | 建议字号 | 字重 | 行高 | 典型场景 |
|---|---:|---:|---:|---|
| H1 页面主标题 | 22px | 600 | 1.25 | “智能体工作台” |
| H2 模块标题 | 17px | 600 | 1.35 | 任务列表、计划、结果文件 |
| H3 重点卡片标题 | 13px | 600 | 1.45 | Plan 步骤标题、当前任务标题 |
| Body 主正文 | 13px | 400–500 | 1.6 | 消息文本、输入内容、文件名 |
| Meta 辅助信息 | 11px | 400–500 | 1.5 | 时间、状态副文案、列表元信息 |
| Tag 标签 | 10px–11px | 500 | 1.4 | 完成、进行中、待执行 |

### 5.2 排版细则

正文区域应优先保证句子连续阅读的舒适度，因此消息文本、输入框文本和步骤说明统一建议使用 **13px** 左右正文，并配合相对宽松的行高。模块标题虽然属于视觉锚点，但不能再回到过大的展示型排版。当前界面的优化方向是“用更小的字号换取更高的信息密度与更成熟的产品感”，后续应继续保持这个方向。

## 6. 栅格、尺寸与间距规范

当前页面属于典型的桌面端三栏工作台结构，其中中栏承担主交互，因此栅格规则必须优先服务中栏宽度。

| 区域 | 建议宽度 | 说明 |
|---|---:|---|
| 左侧任务列表 | 260px | 保持稳定宽度，避免挤压对话区 |
| 中部对话区 | 自适应 | 作为主工作区，优先分配剩余空间 |
| 右侧信息栏 | 340px | 上下堆叠 Plan 与 Result |
| 页面最大宽度 | 1580px | 保持大屏下的稳定边距与视觉收束 |

### 6.1 间距体系

建议沿用 4 的倍数节奏，同时保留少量 10、18、24 这类更适合产品面板的值。

| 间距级别 | 数值 | 推荐用途 |
|---|---:|---|
| XS | 4px | 图标与文字微间距 |
| SM | 8px | 小标签内部、行内元素间距 |
| MD | 12px | 卡片内文本块间距 |
| LG | 16px | 模块内标准间距 |
| XL | 20px | 面板头部与主体分隔 |
| 2XL | 24px | 大卡片圆角与区块外边距 |

## 7. 圆角、边框与阴影规范

当前版本的高级感主要来自“轻边框 + 柔和圆角 + 低对比投影”的组合，而非高强度玻璃拟态或重阴影。后续扩展必须遵守这一原则。

| 元素类型 | 建议圆角 | 边框 | 阴影 |
|---|---:|---|---|
| 顶部总容器 | 24px | 白色透明浅边框 | 柔和大阴影 |
| 主模块面板 | 24px | 浅边框 | 中低强度阴影 |
| 二级卡片 | 18px–20px | 更轻的浅边框 | 极轻阴影或无阴影 |
| 按钮 | 12px | 主按钮可无边框 | 小阴影，仅主按钮使用 |
| 状态标签 | 全圆角 | 细边框 | 通常无阴影 |

在视觉上，边框优先承担“分层”作用，阴影只承担“抬升”作用。若两者同时偏重，界面就会重新变得厚重。设计和开发在落地时应优先减弱阴影，而不是继续堆叠更多发光效果。

## 8. 组件规范

## 8.1 顶部工具栏

顶部工具栏负责提供页面级身份与全局操作。其内容应控制在“品牌名/页面名 + 右侧操作”这一最小集合内，不建议重新引入大段说明文字。工具栏高度应低于主内容区，不应抢夺视觉注意力。

| 元素 | 规范 |
|---|---|
| 品牌标识 | 使用小字号英文标识 + 中文主标题 |
| 次要按钮 | 白底描边，13px 文本 |
| 主按钮 | 品牌蓝底白字，13px 文本 |
| 操作数量 | 建议不超过 2–3 个 |

## 8.2 任务列表

任务列表承担导航与上下文定位功能，应保持简洁、稳定和可快速扫描。选中态的表现应依赖浅色品牌背景、淡蓝边框与轻阴影，而不是高饱和整块填充。

| 状态 | 规范 |
|---|---|
| 默认项 | 浅灰底或透明底，13px 标题 + 11px 元信息 |
| Hover | 边框显现、背景变白 |
| Active | 浅蓝渐变底、浅蓝边框、弱阴影 |
| 元信息 | 使用 11px 灰字，放在标题下方 |

## 8.3 Agent 对话区域

对话区是页面绝对核心，应确保纵向阅读自然、气泡宽度适中、输入区与消息流关系明确。用户消息与 Agent 消息需要通过背景、对齐方式和头部标识来区分，但不应采用过强的对比色造成视觉疲劳。

| 元素 | 规范 |
|---|---|
| 用户消息 | 右对齐、品牌蓝底、白字、20px 圆角 |
| Agent 消息 | 左对齐、浅底卡片、浅边框、上方显示 Agent 名称 |
| 消息正文 | 13px，行高约 1.6 |
| 输入框 | 独立底部区域，轻边框浅底，不与消息区混淆 |
| 输入附属操作 | 仅保留必要图标，如附件、数据、发送 |

### 8.3.1 对话区行为建议

为了让界面更像 Agent 产品而不是普通文件管理页，后续版本建议在不增加复杂度的前提下，为消息区加入时间戳、系统阶段分隔线和执行中状态提示。但这些信息应从属于消息本身，而不是形成额外信息墙。

## 8.4 Plan 计划区域

Plan 区域的核心价值是“解释 Agent 正在做什么”，因此需要比 Result 更具状态感，但信息量不能过载。每个步骤卡片只保留标题、简短描述、时间和状态标签四类信息即可。

| 元素 | 规范 |
|---|---|
| 区域标题 | 17px 标题 + 11px 英文标签 |
| 总体进度 | 细进度条，使用蓝紫到浅桃的渐变 |
| 步骤卡片 | 18px 圆角，浅底，结构紧凑 |
| 步骤标题 | 13px 半粗体 |
| 步骤说明 | 11px 灰字，最多两行 |
| 状态标签 | 10–11px，全圆角，使用语义色 |

## 8.5 Result 结果文件区域

Result 区域强调文件消费效率。文件名必须优先于图标和修饰元素，文件搜索框应轻量而不抢眼。列表项的目标是“可快速浏览和点击”，而不是“像卡片陈列”。

| 元素 | 规范 |
|---|---|
| 搜索框 | 18px 圆角，浅底浅边框，13px 输入字号 |
| 文件项 | 左图标 + 右文件信息，两行结构 |
| 文件名 | 13px 中等字重，超长时单行省略 |
| 元信息 | 11px 灰字，展示类型与大小 |
| Hover | 边框微增强，背景变白 |

## 9. 响应式规范

当前规范优先定义桌面端界面，但在中等宽度和较小桌面中仍需保持四模块结构的可读性。若宽度不足以同时容纳三列，不应简单缩放字体，而应优先通过重排实现稳定阅读。

| 断点 | 建议布局 |
|---|---|
| ≥ 1280px | 标准三栏布局：左任务、中对话、右双面板 |
| 1024px–1279px | 保持三栏，但减少外边距与模块内边距 |
| 768px–1023px | 右侧 Plan/Result 改为折叠或标签切换 |
| < 768px | 采用单列或抽屉式结构，不强行保留三栏并排 |

## 10. 可用性与一致性规则

UI 规范的目标不是仅统一视觉，而是统一使用体验。因此，后续实现中必须遵守以下一致性原则：按钮文案要使用动词开头，状态标签必须语义明确，模块标题不要频繁更名，输入框与发送按钮要始终保留在用户视线可控的位置，所有列表项和文件项的 Hover 反馈都应使用同一套边框与背景提升逻辑。

另外，若未来加入图表预览、参数设置、任务筛选等能力，应优先采用“抽屉、展开态、页内二级面板”这些方式，而不是破坏当前四模块稳定骨架。Ailux Agent 的 UI 应持续强化 **稳定、专业、清晰、可扩展** 的产品印象。

## 11. 设计交付建议

为了便于后续协作，建议设计和前端围绕本规范同步维护一份组件清单，并将以下内容作为最小交付单元：页面结构图、颜色令牌表、字体层级表、按钮状态表、任务项状态表、消息气泡样式表、Plan 卡片样式表和 Result 文件项样式表。这样可以避免未来每次新需求都从视觉判断重新开始。

## 12. 总结

这套 UI 规范以当前 Ailux Agent 四模块界面为基准，核心目的在于把已有的原型样式沉淀为可复用的设计语言。其关键不是追求更复杂的视觉表达，而是通过更稳定的布局、更克制的色彩、更成熟的字号体系和更统一的组件规则，建立一个真正适合 Agent 工作流的产品界面系统。
